# 新需求开发（详细版）

这是你从“想做一个功能”走到“能跑起来”的路线图。

## 最短流程

澄清需求 -> 找入口 -> MVP 方案（最小可用版本） -> 小步实现 -> 最小验证

## 0. 先定风险等级

- 低风险：优先速度，可接受快速试错
- 高风险：必须有测试与回滚策略（出问题时如何撤回）

## 1. 需求澄清

先把需求翻译成“可执行清单”。

输入要点：

- 目标：
- 用户：
- 场景：
- 验收：
- 限制：

你应得到：

- 关键用户流程
- 必做 vs 可延后
- 涉及模块与 API
- 不确定点（尽量少）

## 2. 现状探索

让 AI 给出入口文件、相似实现、数据流路径。

## 2.1 规则与上下文

补充项目约束（命名、目录、接口风格、错误处理）。

## 3. 方案设计

先要 MVP（最小可用版本），再要增强方案。避免一步到位。

## 4. 分步实现

建议顺序：

1. 数据结构或接口
2. 核心逻辑
3. UI / API 接入
4. 验证与日志

## 5. 验证与交付

输出：变更点 + 关键文件 + 验证方式 + 回归点（可能影响的功能）。

## 常见问题

**Q：AI 直接写了一大段代码，怎么办？**
让它先给 MVP（最小可用版本）方案，再小步改动。

**Q：需求不清楚怎么办？**
先把不确定点列出来，确认后再动手。

## 可用提示词

```text
先列出需求的可执行清单和不确定点，再给 MVP（最小可用版本）方案。
```

```text
请按小步实现，每次只改一个点，完成后给出变更说明与验证步骤。
```
