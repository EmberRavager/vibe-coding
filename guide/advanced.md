# 进阶用法（规则与上下文）

当项目开始变长，AI 的“快”会带来副作用。这个章节是刹车和方向盘。

## 1. 规则文件（让 AI 有边界）

你可以把规则理解为“项目的说明书”，AI 写代码前必须先读它。

建议写清：

- 目录结构
- 命名规范
- 错误处理与日志规范
- 测试最低要求

提示词：

```text
请根据本项目风格写一份规则清单，包含目录结构、命名、错误处理与测试要求。
```

## 2. 上下文压缩（避免跑偏）

当对话太长，AI 会忘记关键信息。重新开线程，只保留目标与关键文件。

提示词：

```text
只基于以下关键文件分析：XXX、YYY。忽略其他内容。
```

## 3. 小步提交与回滚

每完成一个小步骤就提交，出错就回滚（退回到上一次正常状态）。这样你永远有可控的起点。

提示词：

```text
请把改动拆成最小可提交步骤。
```

## 4. 让 AI 自检

让 AI 解释自己写的代码，能逼出隐藏假设与边界问题。

提示词：

```text
请逐行解释你刚才的实现，并指出可能的边界风险。
```

## 5. 一键启动与部署脚本

当团队成员多、环境复杂时，一键脚本能显著降低沟通成本。

建议包含：

- 本地启动（安装依赖 + 启动服务）
- 必要环境变量检查
- 最小健康检查

提示词：

```text
请为本项目生成一键启动脚本，包含依赖安装、环境变量检查、服务启动与最小健康检查。
```

## 常见问题

**Q：规则写多了会不会拖慢开发？**
不会。规则是一次性成本，能减少反复返工。

**Q：上下文压缩是不是会丢信息？**
会，但这是有意的取舍。保留最关键的，避免噪声。
