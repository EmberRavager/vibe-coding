# 从 0 到 1 的完整项目路线图

这是一条能走通的路线。你可以照着做一遍，然后把内容替换成自己的项目。

## 第 1 步：一句话说清目标

写下你要做什么、给谁用、做到什么程度算成功。

示例：

“给电商运营做一个订单导出工具，能在 1 分钟内导出 CSV。”

## 第 2 步：找入口与相似实现

目的：让 AI 知道从哪里开始。

提示词：

```text
先找入口和相似实现，给文件路径与理由。
```

## 第 3 步：要 MVP（最小可用版本）方案

目的：先做最小可用版本，再扩展。

提示词：

```text
给 MVP（最小可用版本）方案，别改代码。
```

## 第 4 步：小步实现

目的：一次只改一个点，避免失控。

提示词：

```text
按小步实现，一次只改一个点。
```

## 第 5 步：最小验证

目的：每步都确认“走对了”。

提示词：

```text
给最小验证步骤。
```

## 第 6 步：回归检查（确认没影响其他功能）

目的：确认没有影响其他功能。

提示词：

```text
列出本次改动的回归点（可能影响的功能）。
```

## 第 6.1 步：启动与运行

目的：把功能真正跑起来。

做法：先找项目自带的启动说明（`README.md`、`package.json`、`Makefile`）。

提示词：

```text
请告诉我这个项目如何启动与运行，并给出最小可验证步骤。
```

如果项目启动复杂，可以让 AI 生成一键脚本：

```text
请为本项目生成一键启动脚本，包含依赖安装、环境变量检查、服务启动与最小健康检查。
```

## 第 7 步：记录与复盘

目的：下一次更快，少走弯路。

你可以记录：

- 做了什么
- 为什么这样做
- 如何验证
- 哪些地方容易出错

## 常见问题

**Q：我没有技术背景怎么办？**
只要能说清楚“目标 + 验收”，就可以开始。

**Q：AI 给的方案太大怎么办？**
让它先给 MVP（最小可用版本），再分小步。
